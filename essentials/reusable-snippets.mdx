One of the core principles of software development is DRY (Don't Repeat Yourself). This principle applies equally to documentation. If you find yourself copying the same warning, prerequisite list, or installation steps into multiple pages, create a snippet instead. When you update the snippet, every page that uses it updates automatically.

## How snippets work

A snippet is an `.mdx` file stored in the `snippets/` directory. Mintlify treats every file in that directory as a reusable component — it will not render snippet files as standalone pages in your docs.

To use a snippet, import it at the top of your destination page and render it as a JSX component:

```mdx
import MySnippet from '/snippets/my-snippet.mdx';

<MySnippet />
```

<Note>
Snippet files must live inside the `snippets/` directory. Imports from other directories will not work. The `snippets/` directory itself does not appear in your site navigation.
</Note>

## Creating a snippet

Create a new `.mdx` file in the `snippets/` directory. Write whatever content you want to reuse:

```mdx snippets/prerequisites.mdx
Before you begin, make sure you have:

- Node.js 18 or later installed
- A Mintlify account
- The Mintlify CLI: `npm i -g mint`
```

Then import and use it in any page:

```mdx essentials/quickstart.mdx
---
title: Quickstart
description: Get your docs site running in minutes.
---

import Prerequisites from '/snippets/prerequisites.mdx';

## Before you begin

<Prerequisites />

## Step 1: Initialize your project
...
```

## Passing variables to snippets

Snippets can accept props, letting you customize content without duplicating the snippet file. Use curly braces to reference prop values inside the snippet:

<Steps>
  <Step title="Define the snippet with a variable">

```mdx snippets/api-note.mdx
The `{endpoint}` endpoint requires an API key. Pass it in the `Authorization` header as a Bearer token.
```

  </Step>
  <Step title="Import and pass the prop">

```mdx essentials/api-reference.mdx
---
title: API reference
---

import ApiNote from '/snippets/api-note.mdx';

## Create a user

<ApiNote endpoint="POST /users" />

## Delete a user

<ApiNote endpoint="DELETE /users/{id}" />
```

  </Step>
</Steps>

## Reusable variables

Export JavaScript variables from a snippet file and import them by name. This is useful for version numbers, product names, or any value that appears in many places:

<Steps>
  <Step title="Export variables from the snippet">

```mdx snippets/product-info.mdx
export const version = '2.4.0';
export const cliCommand = 'mint';
export const supportEmail = 'support@example.com';
```

  </Step>
  <Step title="Import the variables into your page">

```mdx index.mdx
---
title: Home
---

import { version, cliCommand, supportEmail } from '/snippets/product-info.mdx';

Welcome to the docs for version **{version}**.

Run `{cliCommand} dev` to start the local preview server.

For help, contact us at {supportEmail}.
```

  </Step>
</Steps>

## Reusable components

For more advanced cases, export an arrow function component from a snippet. This lets you compose HTML layouts that accept props:

<Steps>
  <Step title="Define the component in the snippet">

```mdx snippets/callout-card.mdx
export const CalloutCard = ({ title, description }) => (
  <div style={{ border: '1px solid #e5e7eb', borderRadius: '0.5rem', padding: '1rem' }}>
    <strong>{title}</strong>
    <p>{description}</p>
  </div>
);
```

<Warning>
MDX syntax (like `**bold**` or `<Note>`) does not compile inside arrow function bodies. Use HTML elements like `<strong>`, `<p>`, and `<em>` inside components, or switch to a default export if you need MDX.
</Warning>

  </Step>
  <Step title="Import and render the component">

```mdx guides/features.mdx
---
title: Features
---

import { CalloutCard } from '/snippets/callout-card.mdx';

<CalloutCard
  title="Zero configuration"
  description="Mintlify works out of the box with sensible defaults for every setting."
/>

<CalloutCard
  title="Hot reload"
  description="Changes to your MDX files appear instantly in the local preview."
/>
```

  </Step>
</Steps>

## Best practices

<AccordionGroup>
  <Accordion title="Keep snippets focused">
    Each snippet should contain one logical piece of content — a prerequisite list, a warning about a specific behavior, or a version badge. Avoid creating large all-purpose snippets that are hard to update without unintended side effects.
  </Accordion>
  <Accordion title="Name snippets descriptively">
    Use names that describe what the snippet contains, not where it is used. `prerequisites.mdx` is better than `quickstart-top.mdx` because the same prerequisites might appear on several pages.
  </Accordion>
  <Accordion title="Prefer default exports for MDX content">
    If your snippet contains MDX (Mintlify components, formatted text, headings), use a default export — that is, just write the content in the file body. Reserve arrow function components for pure HTML layouts.
  </Accordion>
  <Accordion title="Use variables for values that change">
    Product version numbers, CLI command names, and support contact details change over time. Export them as named variables from a single snippet so you only update one file when they change.
  </Accordion>
</AccordionGroup>
